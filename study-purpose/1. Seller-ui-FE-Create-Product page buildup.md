### The `useForm()` Hook — What It Returns

```tsx
const {
    register,       // Connects a simple <input> to the form
    handleSubmit,   // Wraps onSubmit — runs validation first
    control,        // Passed to <Controller> for complex components
    setValue,       // Manually set a field value (used for images)
    formState: { errors },  // Object holding all validation errors
} = useForm();
```

#### Plain English:
| Return Value | What It Does | Used By |
|---|---|---|
| `register` | "Hey form, track this `<input>` for me" | `Input` components (title, description, etc.) |
| `handleSubmit` | "When user clicks submit, validate everything first, then call my function" | The `<form>` tag |
| `control` | "Here's a remote control for the form — use it in `<Controller>` wrappers" | `ColorSelector`, `CustomSpecifications`, `CustomProperties` |
| `setValue` | "I want to manually push data into a form field" | `handleImageChange` (images aren't a normal input) |
| `errors` | "Here's every validation error, keyed by field name" | Every component shows its own error message |

### Why `register` vs `control`?

```
register  →  For SIMPLE native HTML inputs (<input>, <textarea>)
              You just spread it: {...register("fieldName")}
              
control   →  For COMPLEX components that aren't a single <input>
              You wrap with <Controller> and manually call field.onChange()
```

### Image State Management

```tsx
const [openImageModal, setOpenImageModal] = useState(false);
const [images, setImages] = useState<(File | null)[]>([null]);
```

> - `openImageModal` — Boolean for a future AI-enhancement modal (not yet wired).
> - `images` — An array that starts as `[null]`. Each slot is either a `File` object or `null` (empty placeholder).

**Why start with `[null]`?** Because we always want at least one upload placeholder visible.

### `handleImageChange` — When User Picks an Image

```tsx
const handleImageChange = (file: File | null, index: number) => {
    const updated = [...images];          // 1. Clone the array (don't mutate state)
    updated[index] = file;                // 2. Put the file in the right slot
    if (index === images.length - 1 && images.length < 8) {
        updated.push(null);               // 3. If they filled the LAST slot, add a new empty one
    }                                     //    (max 8 images)
    setImages(updated);                   // 4. Update React state
    setValue("images", updated);          // 5. Push into react-hook-form so it's in the submit data
};
```

**Visual**:
```
Start:     [null]                          → 1 placeholder shown
Upload 0:  [File, null]                    → image + new placeholder
Upload 1:  [File, File, null]              → 2 images + placeholder
...
Upload 7:  [File, File, ..., File]         → 8 images, no more placeholders (max reached)
```

### `handleRemoveImage` — When User Clicks ✕

```tsx
const handleRemoveImage = (index: number) => {
    setImages((prev) => {
        let updated = [...prev];
        if (index === -1) {
            updated = [null];             // index -1 = "reset everything"
        } else {
            updated.splice(index, 1);     // Remove the image at this position
        }
        if (!updated.includes(null) && updated.length < 8) {
            updated.push(null);           // Ensure there's always an empty slot
        }
        return updated;
    });
};
```

## 📄 FILE 2: `ImagePlaceholder` — The Upload Widget

**Location**: `apps/seller-ui/src/shared/components/image-placeholder/index.tsx`

### TypeScript Interface (The Props Contract)

```tsx
interface ImagePlaceholderProps {
    size: string;                                          // Display text like "765 x 850"
    small?: boolean;                                       // true = 100×100px thumbnail
    onImageChange: (file: File | null, index: number) => void;  // Callback when file picked
    onRemove?: (index: number) => void;                    // Callback when ✕ clicked
    defaultImage?: string | null;                          // Pre-loaded image URL
    index?: number;                                        // Which slot (0, 1, 2...)
    setOpenImageModal?: (open: boolean) => void;           // Toggle AI modal
}
```

> **What is an interface?** Think of it as a **contract**. It says: "If you use this component, you MUST pass these props (unless marked with `?` which means optional)."

### How the Upload Works

```
User clicks the box
  → onClick fires → document.getElementById(inputId)?.click()
    → This programmatically clicks the hidden <input type="file">
      → Browser opens file picker
        → User picks an image
          → handleFileChange fires
            → Creates a blob URL for preview
            → Calls onImageChange(file, index) → goes back to page.tsx
```

### Key Code Explained

```tsx
const [imagePreview, setImagePreview] = useState<string | null>(defaultImage);
```
> Local state holding the preview URL. `<string | null>` is TypeScript saying "this is either a string or null".

```tsx
const handleFileChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const file = e.target.files?.[0];     // Get the first selected file
    if (file) {
        const url = URL.createObjectURL(file);  // Create a temporary browser URL
        setImagePreview(url);                    // Show it in the UI
        onImageChange(file, index);              // Tell parent about it
    }
};
```

```tsx
useEffect(() => {
    return () => {
        if (imagePreview && imagePreview.startsWith("blob:")) {
            URL.revokeObjectURL(imagePreview);   // Cleanup: free memory when unmounted
        }
    };
}, [imagePreview]);
```
> **Memory management**. `URL.createObjectURL` allocates memory. When the component is removed from the page, this cleanup function releases it. Without this, you'd have a **memory leak**.

## 📄 FILE 3: `Input` — Reusable Form Field

**Location**: `packages/components/input/index.tsx`

### The TypeScript Gymnastics Explained

```tsx
interface BaseProps {
    label?: string;
    type?: "text" | "email" | "password" | "number" | "textarea";
    className?: string;
    error?: string;
    rows?: number;
    cols?: number;
}

type InputProps = BaseProps & React.InputHTMLAttributes<HTMLInputElement>;
type TextareaProps = BaseProps & React.TextareaHTMLAttributes<HTMLTextAreaElement>;
type Props = InputProps | TextareaProps;
```

> **In plain English**:
> - `BaseProps` — Our custom props (label, error message, etc.)
> - `InputProps` — Our props + ALL native `<input>` props (placeholder, onChange, etc.)
> - `TextareaProps` — Our props + ALL native `<textarea>` props
> - `Props` — "This component accepts EITHER input props OR textarea props"

### `forwardRef` — Why?

```tsx
const Input = forwardRef<HTMLInputElement | HTMLTextAreaElement, Props>(
    ({ label, type = "text", className, error, rows, cols, ...props }, ref) => {
```

> `react-hook-form`'s `register()` returns a `ref`. React needs `forwardRef` to pass that ref through to the actual `<input>` DOM element. **Without `forwardRef`, `register()` would silently fail** — the form wouldn't track the field.

### The Rendering Logic

```tsx
{type === "textarea" ? (
    <textarea ref={ref as React.Ref<HTMLTextAreaElement>} ... />
) : (
    <input ref={ref as React.Ref<HTMLInputElement>} ... />
)}
```
> If `type="textarea"`, render a `<textarea>`. Otherwise render an `<input>`. Both get the forwarded ref and all spread props.

```tsx
{error && (
    <span className="text-red-400 text-xs mt-0.5">{error}</span>
)}
```
> If there's a validation error, show it in red below the field.

### How `page.tsx` Uses It

```tsx
<Input
    label="Product Title *"
    placeholder="Enter product title"
    {...register("title", { required: "Product title is required" })}
    error={errors.title?.message as string}
/>
```

**What `{...register("title", {...})}` expands to**:
```tsx
{
    name: "title",
    ref: [internal ref],
    onChange: [internal handler],
    onBlur: [internal handler],
}
```
> These spread onto the `<input>` via `...props`, connecting it to react-hook-form.

## 📄 FILE 4: `ColorSelector` — Pick Product Colors

**Location**: `packages/components/color-selector/index.tsx`

### Why `Controller` Instead of `register`?

Colors aren't a simple text input — they're an **array of hex strings** (`["#FF0000", "#2563EB"]`). You can't use `register()` for that. Instead, you wrap with `<Controller>` which gives you `field.onChange()` to manually push data.

### The Data Flow

```
defaultColors (9 preset colors)
     +
customColors (user-added via color picker)
     ↓
Rendered as clickable color swatches
     ↓
Click toggles the color in/out of field.value (an array)
     ↓
field.onChange([...updated array]) → react-hook-form knows
```

### Key Code

```tsx
const [customColors, setCustomColors] = useState<{ name: string; value: string }[]>([]);
const [showPicker, setShowPicker] = useState(false);
const [newColor, setNewColor] = useState("#ffffff");
```
> Three pieces of local state:
> - `customColors` — Colors the user added manually
> - `showPicker` — Whether the native color picker is visible
> - `newColor` — The current color in the picker (before "Add" is clicked)

```tsx
<Controller
    name="colors"
    control={control}
    defaultValue={[]}
    render={({ field }) => (
```
> `Controller` creates a form field called `"colors"` with default value `[]` (empty array). The `field` object inside `render` has `.value` and `.onChange()`.

### Toggle Logic (Click a Swatch)

```tsx
onClick={() => {
    const current: string[] = field.value || [];
    field.onChange(
        isSelected
```

## 📄 FILE 5: `CustomSpecifications` — Key-Value Pair Rows

**Location**: `packages/custom-specifications/index.tsx`

### Purpose
For product specs like: `Material → Cotton`, `Weight → 250g`, `Dimensions → 10×20cm`.

### `useFieldArray` — The Magic Hook

```tsx
const { fields, append, remove } = useFieldArray({
    control,
    name: "custom_specifications"
});
```

> This hook manages a **dynamic array** inside the form. Think of it like `useState` for an array, but it's wired into react-hook-form.

| Function | What It Does |
|---|---|
| `fields` | The current array of `{name, value}` objects |
| `append({name: "", value: ""})` | Adds a new empty row |
| `remove(index)` | Deletes a row at position `index` |

### Each Row

```tsx
{fields.map((item, index) => (
    <div key={index} className='flex gap-2 items-center'>
        <Controller name={`custom_specifications.${index}.name`} ...>
            <Input label="Specification Name" placeholder="e.g., Material" ... />
        </Controller>
        <Controller name={`custom_specifications.${index}.value`} ...>
            <Input label="Specification Value" placeholder="Enter value" ... />
        </Controller>
        <button onClick={() => remove(index)}>
            <Trash size={20} />       {/* Delete this row */}
        </button>
    </div>
))}
```

> **Note the dynamic field name**: `` `custom_specifications.${index}.name` `` creates names like `custom_specifications.0.name`, `custom_specifications.1.name`, etc. React-hook-form knows how to nest these into an array of objects.

### Add Button

```tsx
<button onClick={() => append({name: "", value: ""})}>
```

## 📄 FILE 6: `CustomProperties` — Key → Multi-Value Builder

**Location**: `packages/components/custom-properties/index.tsx`

### Purpose
For properties like: `Size → [S, M, L, XL]`, `Storage → [128GB, 256GB, 512GB]`.

**Different from CustomSpecifications**: Specs have ONE value per key. Properties have MULTIPLE values per key.

### The Data Shape

```tsx
type Property = {
    label: string;      // "Size", "Storage", etc.
    values: string[];   // ["S", "M", "L", "XL"]
};
```

### State Management

```tsx
const [properties, setProperties] = useState<Property[]>([]);
const [newLabel, setNewLabel] = useState('');
const [newValues, setNewValues] = useState<Record<number, string>>({});
```

> - `properties` — The array of all property groups
> - `newLabel` — Text in the "add property" input
> - `newValues` — A **dictionary** mapping property index → current value input text. `Record<number, string>` means `{ 0: "typing...", 1: "also typing..." }`.

### Why `syncToForm`?

```tsx
const syncToForm = (updated: Property[]) => {
    setProperties(updated);    // Update local state for UI
    field.onChange(updated);    // Push to react-hook-form for submit data
};
```
> This component manages its own state (for the dynamic UI) AND syncs to the form. It's a **dual-state pattern** — local state for rendering, form state for submission.

### The UI Flow

```
Step 1: Type "Size" → Click ➕
┌────────────────────────────────────────┐
│ Size                              ✕    │
│ [chips area - empty]                   │
│ [Add value input...          ] [➕]    │
└────────────────────────────────────────┘

Step 2: Type "S" → press Enter or click ➕
┌────────────────────────────────────────┐
│ Size                              ✕    │
│ [S ✕]                                  │
│ [Add value input...          ] [➕]    │
└────────────────────────────────────────┘

Step 3: Add more values
┌────────────────────────────────────────┐
│ Size                              ✕    │
│ [S ✕] [M ✕] [L ✕] [XL ✕]             │
│ [Add value input...          ] [➕]    │
└────────────────────────────────────────┘

[Add property label...              ] [➕]   ← Add another property
```

### Enter Key Support

```tsx
onKeyDown={(e) => {
    if (e.key === 'Enter') {
        e.preventDefault();        // Don't submit the form!
        handleAddValue(index);     // Add the value instead
    }
}}
```
> Without `e.preventDefault()`, pressing Enter inside any input would submit the entire form. This intercepts it.

## 🔑 TypeScript Concepts Used (Quick Reference)

| Concept | Example | Meaning |
|---|---|---|
| `interface` | `interface ImagePlaceholderProps {...}` | Defines the shape of an object (like a contract) |
| `?` after prop name | `small?: boolean` | This prop is optional |
| `<Type>` generics | `useState<string \| null>` | "This state holds a string or null" |
| `as` type assertion | `errors.title?.message as string` | "Trust me, this value is a string" |
| `?.` optional chaining | `e.target.files?.[0]` | "If files exists, get index 0, otherwise undefined" |
| `Record<K,V>` | `Record<number, string>` | A dictionary: keys are numbers, values are strings |
| Union types | `"text" \| "email" \| "textarea"` | "One of these values" |
| `forwardRef` | `forwardRef<HTMLInputElement, Props>` | Lets a parent pass a `ref` through to a child DOM element |
| `& (intersection)` | `BaseProps & React.InputHTMLAttributes` | "Combine both types into one" |

## 💡 Key Takeaways

1. **`page.tsx` is the orchestrator** — it creates the form, manages image state, and passes everything down to children.
2. **`react-hook-form` is the central nervous system** — every piece of form data flows through it.
3. **`register` for simple inputs, `control` for complex ones** — this is the most important pattern to remember.
4. **Components are dumb** — they don't know about the form. They just receive props and call callbacks. This makes them reusable.
5. **`forwardRef` exists because of `register`** — without it, react-hook-form can't attach its ref to the DOM input.
6. **`useFieldArray` manages dynamic lists** — it's like `useState` for arrays but wired into the form.
7. **`syncToForm` dual-state pattern** — some components manage their own visual state AND sync to the form.
